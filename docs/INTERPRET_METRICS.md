# Interpret Metrics — Calculation and Validation

v16 changes the Interpret metrics from static demo values into **run-aware metrics**.

The important distinction is:

- **Citation support rate**, **unsupported claim rate** and **human review rate** are calculated from the actual current run: generated output + retrieved source chunks.
- **Obligation precision** and **obligation recall** require an optional reviewer-written gold obligations file for the same document. If no gold set is uploaded, they are shown as **N/A** rather than using synthetic demo numbers.

This keeps the evaluation honest: the app can measure textual support on every run, but it cannot know true precision/recall without ground truth.

---

## 1. Obligation precision

**Question answered:** Of the obligations generated by the model, how many match independently expected obligations?

**Formula:**

```text
matched generated obligations / generated obligations
```

**Example:**

```text
8 matched generated obligations / 9 generated obligations = 89%
```

**Evidence required:** A reviewer uploads a gold obligations JSON/CSV file for the same document.

**Validation method:**

1. The app extracts generated obligations from `obligation_extraction` and `traceability_matrix`.
2. The reviewer-provided gold set is loaded.
3. Each generated obligation is compared against gold obligations using rule-based similarity and required-term overlap.
4. A generated obligation is counted as matched if it passes the configured threshold.

**Important limitation:** This is only meaningful if the gold set was written independently of the model output.

---

## 2. Obligation recall

**Question answered:** Of the expected obligations in the gold set, how many did the model recover?

**Formula:**

```text
matched gold obligations / total gold obligations
```

**Example:**

```text
8 matched gold obligations / 10 gold obligations = 80%
```

**Evidence required:** A reviewer uploads a gold obligations JSON/CSV file for the same document.

**Validation method:**

1. Each gold obligation is compared to all generated obligations.
2. The best generated match is selected.
3. The gold obligation is counted as found if the best match passes the threshold.

**Important limitation:** Recall cannot be calculated from the generated output alone. Without gold obligations, the app displays **N/A**.

---

## 3. Citation support rate

**Question answered:** How many generated claims have an existing citation and enough textual support in the cited source chunk?

**Formula:**

```text
supported cited claims / checked claims
```

**Example:**

```text
8 supported claims / 9 checked claims = 89%
```

**Evidence required:** No gold set is required. This is calculated from the actual current run.

**Validation method:**

For each extracted claim/citation, the app checks:

- whether the cited source ID exists, such as `[S1]`;
- whether the proposed quote appears in the source or a strong source sentence can be found;
- token overlap;
- n-gram similarity;
- number consistency;
- modality and negation mismatches.

A claim is counted as supported if the support score is at or above the configured support threshold.

**Important limitation:** Textual support is not legal interpretation. It only shows whether the claim appears grounded in the source text.

---

## 4. Unsupported claim rate

**Question answered:** How many generated claims are missing a source or have very weak textual support?

**Formula:**

```text
unsupported claims / checked claims
```

**Example:**

```text
1 unsupported claim / 9 checked claims = 11%
```

**Evidence required:** No gold set is required. This is calculated from the actual current run.

**Validation method:**

A claim is counted as unsupported when:

- the citation source ID does not exist;
- the support score is below the unsupported threshold;
- the claim contains unsupported numbers, modal strength or wording not found in the source.

**Important limitation:** A low unsupported rate is encouraging but still not proof of compliance correctness.

---

## 5. Human review rate

**Question answered:** How many claims should be reviewed by a human before being trusted?

**Formula:**

```text
human-review claims / checked claims
```

**Example:**

```text
1 human-review claim / 9 checked claims = 11%
```

**Evidence required:** No gold set is required. This is calculated from the actual current run.

**Validation method:**

The app routes a claim to human review when it detects:

- missing source;
- weak support;
- proposed quote not present in source;
- number mismatch;
- possible negation/modality issue;
- model-generated review reason.

**Important limitation:** In regulated workflows, a higher human review rate is not always bad. It can mean the system is surfacing uncertainty rather than hiding it.

## Gold obligations file format

JSON format:

```json
{
  "name": "Run-specific gold obligations for this document",
  "gold_obligations": [
    {
      "id": "G1",
      "obligation": "Customer tax identification number must be present before SAF-T export.",
      "required_terms": ["customer tax identification", "SAF-T export"],
      "expected_source": "S1",
      "expected_quote": "Invoices included in a Portuguese SAF-T export must include the customer tax identification number."
    }
  ]
}
```

CSV format:

```text
id,obligation,required_terms,expected_source,expected_quote
G1,Customer tax identification number must be present before SAF-T export.,customer tax identification; SAF-T export,S1,Invoices included in a Portuguese SAF-T export must include the customer tax identification number.
```

---

## Recommended validation workflow

1. Upload or paste the compliance document.
2. Generate the Interpret package.
3. Review the source quote matrix and citation support table.
4. Upload a gold obligations file written independently of the model output.
5. Re-check the Quality Metrics tab.
6. Export the metrics CSV and human review queue.
7. Use the output as review evidence, not as automatic compliance proof.

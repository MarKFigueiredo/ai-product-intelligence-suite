# UX Review Density Policy

This document defines how the portfolio balances a simple first-click review path with deeper evidence for technical reviewers.

## Purpose

The application contains substantial evidence, traceability, quality controls and review artifacts. The first user experience should still remain lightweight.

The default experience should help a reviewer understand the hero workflow quickly, while keeping deeper evidence available on demand.

## Design rule

Default UI = compact decision path.

Expanders and supporting documents = deeper evidence and technical review.

## What should be visible first

The first review path should show:

- the problem being addressed;
- the hero compliance-to-product workflow;
- a short example of source-to-output transformation;
- human review boundaries;
- QA and release-risk controls;
- audit/evidence output.

## What can stay deeper

The following can be available behind expanders or supporting documents:

- full evidence tables;
- lineage details;
- gate details;
- raw intermediate artifacts;
- extended validation notes;
- companion-domain material.

## Review principle

The interface should reduce first-click cognitive load without hiding the product depth.

The goal is to make the case study understandable quickly, while preserving enough detail for product, engineering, QA, compliance and audit review.
